# Neuron

## Register

product

## Users

Neuron is for developers, technical writers, and knowledge workers who keep durable notes as local Markdown or MDX files. They use it at a desktop for sustained writing, cross-referencing, and review, often with the keyboard and often for long sessions.

## Product Purpose

Neuron turns a local notes directory into a fast authoring workspace: browse files, edit Markdown and MDX, inspect the rendered result, filter by tags, and understand relationships through wiki-links. Success means users can move from thought to saved file without losing context or wondering whether the application changed their source of truth.

## Brand Personality

Precise, calm, and capable. The interface should feel engineered rather than decorated: compact where repetition benefits from density, generous where reading needs focus, and quietly responsive to every action.

## Anti-references

- Not a marketing site compressed into an app shell.
- Not a neon cyberpunk terminal or a wall of glowing telemetry.
- Not a soft, rounded productivity toy built from nested cards.
- Not a browser-first responsive layout with mobile navigation patterns.
- Not an IDE clone that exposes complexity before it is useful.

## Design Principles

1. Keep the file visible. The current note, its save state, and its relationship to the workspace should never be ambiguous.
2. Preserve writing momentum. Common actions stay close, keyboard focus remains obvious, and state changes complete without ceremony.
3. Separate tools from content. Navigation, source, preview, and graph each have a distinct spatial role with minimal visual containers.
4. Let density follow frequency. Repeated lists are compact; prose and error explanations are given room.
5. Prefer honest local behavior. Use direct language about saving, parsing, empty results, and file operations.

## Accessibility & Inclusion

Target WCAG 2.2 AA for renderer UI. All actions require visible keyboard focus and accessible names; text and controls must meet contrast requirements; color never carries state alone; reduced-motion preferences disable nonessential transitions; destructive actions require explicit confirmation; and empty, error, and saving states must be announced in text.
